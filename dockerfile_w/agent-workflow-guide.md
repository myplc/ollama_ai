# 🤖 AI 코딩 에이전트 워크플로우 가이드

> 이 문서는 AI 코딩 에이전트(Antigravity + Claude 등)가 실제로 코드를 수정하고,
> 실행하고, 피드백을 받아 반복 개선하는 **전체 작업 흐름**을 정리한 것입니다.
> 다른 에이전트가 이 가이드를 참고하면 동일한 수준의 체계적인 작업이 가능합니다.

---

## 1. 세 가지 모드: PLANNING → EXECUTION → VERIFICATION

에이전트의 모든 작업은 이 세 모드를 순환합니다.

```
┌──────────┐     ┌───────────┐     ┌──────────────┐
│ PLANNING │ ──▶ │ EXECUTION │ ──▶ │ VERIFICATION │
│ (조사/설계)│     │ (구현/수정) │     │  (테스트/검증) │
└──────────┘     └───────────┘     └──────────────┘
      ▲                                    │
      └────────────────────────────────────┘
              문제 발견 시 되돌아감
```

### 1.1 PLANNING (계획)

**목표**: 코드를 한 줄도 고치기 전에 충분히 이해하고 설계한다.

| 단계 | 행동 | 도구 |
|------|------|------|
| ① 요구사항 파악 | 사용자 요청을 정확히 분석 | — |
| ② 코드베이스 탐색 | 관련 파일/구조를 파악 | `list_dir`, `find_by_name`, `grep_search` |
| ③ 기존 코드 분석 | 수정할 코드의 문맥을 이해 | `view_file` |
| ④ 구현 계획 작성 | 어떤 파일을 어떻게 고칠지 계획 | `write_to_file` (implementation_plan.md) |
| ⑤ 사용자 승인 요청 | 계획을 보여주고 동의를 구함 | `notify_user` |

**핵심 원칙**:
- **절대 코드를 먼저 고치지 않는다.** 먼저 충분히 읽고 이해한다.
- 파일을 처음 열 때는 **최대한 넓게(800줄)** 읽어 전체 맥락을 파악한다.
- 검색할 때는 `grep_search`와 `find_by_name`을 **병렬로** 호출해 시간을 아낀다.
- 과거 대화 기록이나 KI(Knowledge Item)가 있으면 **반드시 먼저 확인**한다.

### 1.2 EXECUTION (실행)

**목표**: 계획대로 코드를 수정하고, 필요 시 실행한다.

| 단계 | 행동 | 도구 |
|------|------|------|
| ① 파일 수정 | 한 곳만 고치면 `replace_file_content`, 여러 곳이면 `multi_replace_file_content`, 새 파일은 `write_to_file` | 코드 편집 도구 |
| ② 빌드/실행 | 수정한 코드가 돌아가는지 확인 | `run_command` |
| ③ 에러 확인 | 터미널 출력을 읽고 에러를 분석 | `command_status`, `read_terminal` |
| ④ 즉시 수정 | 에러가 있으면 바로 고침 | 코드 편집 도구 |

**핵심 원칙**:
- 파일 수정 시 **정확한 줄 번호와 기존 코드**를 지정해야 한다. 직접 `view_file`로 확인한 후 수정한다.
- 하나의 파일에 대해 **동시에 두 편집 도구를 호출하지 않는다.**
- 긴 명령은 백그라운드로 보내고 `command_status`로 주기적으로 확인한다.
- 서버 같은 장기 실행 프로세스는 `WaitMsBeforeAsync`를 짧게(500ms 등) 설정하고 별도로 상태를 확인한다.

### 1.3 VERIFICATION (검증)

**목표**: 수정이 의도대로 동작하는지 확실히 확인한다.

| 단계 | 행동 | 도구 |
|------|------|------|
| ① 자동 테스트 | 유닛 테스트, 린트 실행 | `run_command` |
| ② 브라우저 확인 | UI 변경이면 실제 브라우저에서 확인 | `browser_subagent` |
| ③ 결과 문서화 | 무엇을 했고 어떻게 검증했는지 기록 | `write_to_file` (walkthrough.md) |
| ④ 사용자 보고 | 결과를 요약해서 알림 | `notify_user` |

## 2. 파일 탐색의 기술

### 2.1 프로젝트 구조 파악

```
# 1단계: 루트 디렉토리 확인
list_dir("/project/root")

# 2단계: 특정 파일 유형 검색
find_by_name(SearchDirectory="/project", Pattern="*.js", Type="file")

# 3단계: 코드 내 키워드 검색
grep_search(SearchPath="/project/src", Query="handleSubmit")
```

### 2.2 효율적인 검색 전략

- **파일 찾기**: `find_by_name` → 파일명/확장자 기반
- **코드 찾기**: `grep_search` → 함수명, 변수명, 에러 메시지 기반
- **구조 파악**: `list_dir` → 디렉토리 트리 확인
- **병렬 호출**: 서로 의존성 없는 검색은 반드시 **동시에** 실행

```
# 좋은 예: 병렬로 동시 검색
grep_search(Query="login", SearchPath="/src")        # 동시 실행
find_by_name(Pattern="auth*", SearchDirectory="/src") # 동시 실행

# 나쁜 예: 순차 실행 (불필요하게 느림)
grep_search(Query="login", ...)  # 먼저 실행 후
find_by_name(Pattern="auth*", ...)  # 결과 기다린 다음 실행
```

## 3. 파일 수정의 정석

### 3.1 수정 도구 선택 기준

| 상황 | 사용할 도구 |
|------|------------|
| 새 파일 생성 | `write_to_file` |
| 한 곳만 수정 (연속된 블록) | `replace_file_content` |
| 여러 곳 수정 (떨어진 블록들) | `multi_replace_file_content` |
| 파일 전체 덮어쓰기 | `write_to_file` (Overwrite=true) |

### 3.2 수정 전 반드시 할 것

```
# 1. 먼저 해당 파일의 현재 내용을 확인
view_file(AbsolutePath="/project/src/app.js")

# 2. 정확한 줄 번호를 확인한 후 수정
replace_file_content(
    TargetFile="/project/src/app.js",
    StartLine=42,
    EndLine=45,
    TargetContent="기존 코드 정확히 복사",
    ReplacementContent="새로운 코드"
)
```

### 3.3 흔한 실수와 방지법

> [!CAUTION]
> - `TargetContent`의 **들여쓰기(공백)가 정확히** 일치해야 한다
> - 같은 파일에 **두 편집 도구를 동시에 호출하면 안 된다**
> - 수정 범위가 넓으면 `view_file`로 **반드시 줄 번호를 재확인**한다

## 4. 명령어 실행과 피드백 루프

### 4.1 기본 실행 패턴

```
# 짧은 명령: 동기적으로 실행 (WaitMs 크게)
run_command(CommandLine="npm test", Cwd="/project", WaitMsBeforeAsync=10000)

# 긴 명령/서버: 비동기로 실행 (WaitMs 작게)
run_command(CommandLine="npm run dev", Cwd="/project", WaitMsBeforeAsync=500)
# → 반환된 CommandId로 상태 확인
command_status(CommandId="xxx", WaitDurationSeconds=5)
```

### 4.2 에러 발생 시 피드백 루프

```
┌─────────────┐
│  코드 수정    │
└──────┬──────┘
       ▼
┌─────────────┐     에러 없음    ┌──────────┐
│ 실행/테스트   │ ─────────────▶ │ 검증 완료  │
└──────┬──────┘               └──────────┘
       │ 에러 발생
       ▼
┌─────────────┐
│ 에러 메시지   │
│ 분석          │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 원인 파악    │ ← grep_search로 관련 코드 추가 조사
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 코드 재수정   │ ──▶ (다시 실행/테스트로 돌아감)
└─────────────┘
```

**핵심**: 에러가 나면 **즉시 수정하고 다시 실행**한다. 이 루프를 문제가 해결될 때까지 반복한다.

## 5. 브라우저 기반 검증 (UI 작업)

UI를 수정했을 때는 반드시 실제 브라우저에서 확인한다.

```
browser_subagent(
    TaskName="UI 변경사항 검증",
    Task="http://localhost:3000 에 접속하여 로그인 페이지의 버튼이 올바르게 표시되는지 확인하고, 
          버튼 클릭 시 정상 동작 여부를 스크린샷과 함께 보고해주세요.",
    RecordingName="login_button_test"
)
```

> [!TIP]
> - `browser_subagent`에게 보내는 Task는 **매우 구체적**으로 작성한다  
> - 확인할 URL, 클릭할 요소, 기대하는 결과를 명확히 기술한다  
> - 서버가 실행 중인지 먼저 확인한 후 브라우저를 띄운다

## 6. 작업 관리: task.md 체크리스트

복잡한 작업은 반드시 `task.md`를 만들어 진행 상황을 추적한다.

```markdown
# 로그인 기능 개선

- [x] 기존 로그인 코드 분석
- [x] 입력 검증 로직 추가
- [/] 에러 메시지 UI 개선  ← 현재 진행 중
- [ ] 비밀번호 재설정 기능
- [ ] 테스트 작성 및 실행
```

| 표시 | 의미 |
|------|------|
| `[ ]` | 아직 안 함 |
| `[/]` | 진행 중 |
| `[x]` | 완료 |

## 7. 사용자와의 소통 원칙

### 7.1 언제 사용자에게 물어보는가

- 요구사항이 **모호하거나 여러 해석이 가능**할 때
- **파괴적인 변경** (파일 삭제, DB 스키마 변경 등)을 하기 전
- 여러 선택지가 있고 **사용자의 선호가 중요**할 때
- 복잡한 계획을 세웠고 **승인이 필요**할 때

### 7.2 보고 형식

```markdown
## 변경 요약
- `auth.js`: 로그인 검증 로직 추가 (L42-58)
- `login.css`: 에러 메시지 스타일 개선

## 테스트 결과
- ✅ npm test: 15/15 통과
- ✅ 브라우저 확인: 로그인 실패 시 에러 메시지 정상 표시

## 확인 필요
- 비밀번호 최소 길이를 8자로 설정했는데, 다른 기준이 있으신가요?
```

## 8. 고급 패턴

### 8.1 병렬 작업으로 속도 높이기

서로 **의존성이 없는 작업**은 반드시 동시에 실행한다.

```
# 좋은 예: 3개 파일을 동시에 읽기
view_file("/src/a.js")  # 동시
view_file("/src/b.js")  # 동시  
view_file("/src/c.js")  # 동시

# 나쁜 예: 하나씩 순서대로 읽기 (3배 느림)
view_file("/src/a.js")  # 기다림
view_file("/src/b.js")  # 기다림
view_file("/src/c.js")  # 기다림
```

### 8.2 에러 디버깅 전략

1. **에러 메시지를 정확히 읽는다** — 파일명, 줄 번호, 에러 타입 파악
2. **해당 코드를 `view_file`로 확인**한다
3. **관련 코드를 `grep_search`로 추가 조사**한다
4. **가설을 세우고 수정**한다 — 가장 가능성 높은 원인부터
5. **수정 후 즉시 재실행**한다
6. **같은 에러가 반복되면 접근 방식을 바꾼다**

### 8.3 대규모 리팩토링

1. **영향 범위 분석**: `grep_search`로 변경할 함수/변수를 사용하는 모든 곳을 찾는다
2. **의존성 순서 파악**: 어떤 파일을 먼저 고쳐야 하는지 파악한다
3. **단위별 수정-테스트**: 한 번에 모든 파일을 고치지 않고, 모듈 단위로 수정→테스트를 반복한다
4. **회귀 테스트**: 전체 테스트를 돌려 다른 곳이 깨지지 않았는지 확인한다

## 9. 안전 수칙

> [!WARNING]
> - 파일 삭제, DB 변경 등 **파괴적 명령**은 반드시 사용자 승인 후 실행
> - `SafeToAutoRun=true`는 **읽기 전용 명령**(ls, cat, echo 등)에만 사용
> - 외부 API 호출, 패키지 설치 등은 **항상 사용자에게 확인**
> - 수정 전 원본을 확인해두고, 실수 시 **되돌릴 수 있도록** 대비

## 10. 전체 워크플로우 요약 (한 눈에 보기)

```
사용자 요청
    │
    ▼
[PLANNING] ─── 코드베이스 탐색 ─── 구현 계획 작성 ─── 사용자 승인
    │
    ▼
[EXECUTION] ─── 파일 수정 ─── 실행/빌드 ─── 에러? ──▶ 수정 후 재실행 (반복)
    │                                         │
    │         ◀── 예상 못한 복잡성 발견 ────────┘ (PLANNING으로 복귀)
    │
    ▼
[VERIFICATION] ─── 테스트 실행 ─── 브라우저 확인 ─── 결과 문서화
    │
    ▼
사용자에게 결과 보고
```

## 부록: 도구 빠른 참조표

| 도구 | 용도 | 예시 |
|------|------|------|
| `list_dir` | 디렉토리 구조 확인 | 프로젝트 레이아웃 파악 |
| `find_by_name` | 파일명/확장자로 검색 | `*.test.js` 파일 찾기 |
| `grep_search` | 코드 내 텍스트 검색 | 함수 사용처 찾기 |
| `view_file` | 파일 내용 읽기 | 수정 전 코드 확인 |
| `write_to_file` | 새 파일 생성 | 컴포넌트 추가 |
| `replace_file_content` | 단일 블록 수정 | 함수 하나 수정 |
| `multi_replace_file_content` | 다중 블록 수정 | 여러 곳 동시 수정 |
| `run_command` | 터미널 명령 실행 | `npm test`, `node app.js` |
| `command_status` | 실행 중인 명령 확인 | 서버 로그 확인 |
| `browser_subagent` | 브라우저 조작 | UI 테스트, 스크린샷 |
| `search_web` | 웹 검색 | API 문서, 에러 해결법 |
| `notify_user` | 사용자 알림 | 결과 보고, 질문 |
